# NekoCode 大项目分析能力增强方案

## 问题

NekoCode 分析大项目时存在两个层面缺陷：
- **提示词层面**：缺乏编排思维，模型倾向于过度探索而非高效行动，没有 token 成本意识
- **架构层面**：缺少静态预分析、工具结果缺乏智能拦截、消息切片不够精准

本方案引入 5 个核心机制：
- Orchestration Instructions（Manifest-First 探索策略 + 违规意识）
- Tool Constraint Prompting（动态配额 + 阶梯税率 + 申请扩展）
- Exploration Score（衰减分探索槽，替代死轮次止损）
- Tool-Result Interceptor（修复预算 + 去重 + 大文件强制分页）
- ProjectIndex（静态分析预计算）

---

## 一、Orchestration Instructions — 编排思维框架

### 1.1 Manifest-First 探索策略

分析大项目时，模型最容易犯的错误是一头扎进源码逐行读。需要在提示词顶部植入行为框架，训练它"先看地图、再看细节"：

> 分析项目时，不要直接读源码。顺序如下：
> 1. 先扫描 `go.mod`/`package.json`/`Cargo.toml` 了解依赖和项目类型
> 2. 读取 `NECOCODE.md`/`README.md` 了解项目约定和架构描述
> 3. 用 `list`/`glob` 建立目录骨架，理解模块分布
> 4. 用 `grep` 找关键符号的跨文件引用关系
> 5. 最后才 `read` 具体文件——且只读你需要的那部分

目标不是记住每一行的实现，而是理清模块间的 **关系和依赖**。

### 1.2 无节制读取 = 严重违规

提示词中把超量读取定义为违规行为：

> 一次 read 读取超过 1000 行源码且无结构性理由，视为严重上下文预算违规。读大文件时必须指定 `startLine`/`endLine` 只看关键段落，全文通读基本不需要。

**注意：口头警告必须配合代码强制。** 提示词建立羞耻感是"软约束"，Interceptor 的强制分页是"硬约束"。软硬结合才能生效——模型知道即使它自我欺骗编造了"结构性理由"，客户端也会截断。

## 二、Tool Constraint Prompting — 动态配额系统

### 2.1 核心概念：单轮预算窗口（Rolling Turn Window）

不给整个会话设置死配额，而是每轮开始时根据当前上下文水位动态计算。水位用占比而非绝对值，适配不同大小的 token 预算。

### 2.2 三档阶梯税率

| 水位 | 占比 | Read | Grep | 模式 |
|------|------|------|------|------|
| 绿区 | < 40% | 5 | 5 | 软建议，措辞柔和，不拦截 |
| 黄区 | 40%-80% | 3 | 2 | 硬拦截，超额直接报错 |
| 红区 | > 80% | 1 | 1 | **强制 MicroCompact 释放空间；连续 2 轮无效则熔断触发强制沉淀** |

绿区上下文宽松，允许充分探索。黄区开始承压，强制模型精准下刀——每轮只有 3 次 read、2 次 grep，超额拒绝执行。**红区熔断——Compact 边际效益递减检测：** 存在一种极端死锁：分析巨大函数或海量遗留系统时，当前轮次刚刚读取的必要上下文本身就超过了预算的 80%。MicroCompact 只能清理旧历史，对这些"最新轮次的硬核干货"无能为力。模型会卡在"水位 > 80% → 触发 Compact → 释放不了空间 → 下一轮仍 > 80% → 再次 Compact"的无限循环中。

**熔断条件：** 连续 2 轮触发 MicroCompact 后，Token 水位下降比例 < 5%——说明上下文已压缩到极限，全是不可抛弃的硬核干货。此时自动触发 **3.4 强制沉淀**，将当前轮次之前的所有线索强制提炼为 `<module-summary>`，物理清空历史，只保留最后 1 轮消息和结构化总结，强行把水位砸回绿区。

### 2.3 配额扩展机制（Quota Extension）

硬配额可能误伤真正复杂的任务。在系统提示词中告知模型：如果配额耗尽但确实需要更多读取来定位问题，可以输出自然语言申请（如"我需要申请额外配额来检查核心逻辑"），系统解析器会识别扩展意图并重置本轮配额。触发靠语义理解而非文本匹配——防止模型机械复制示例格式。

约束：每轮最多扩展 2 次，防止滥用。

### 2.4 反滥用：Quota Request 附加扣分

模型一旦学会无限申请扩展，配额系统就形同虚设。每次成功申请配额扩展时，**直接从 Exploration Score 扣除 30 分**。这意味着申请配额是有代价的——你获得了更多读取权，但更快逼近强制沉淀。模型必须权衡"我真的需要再看吗"vs"看了之后分数会不会扣光"。

### 2.5 拦截报错必须携带行动指引（防死磕）

黄区/红区拒绝超额请求时，返回的报错信息不能只是生硬的"配额耗尽，请求被拒绝"。某些模型有"强迫症"——收到拒绝后会在下一轮固执地重新发起同一个 read，导致跟拦截器反复死磕，白白浪费 token。

拒绝报错必须包含明确的下一步行动指引：

> [配额拦截] 本轮 Read 配额已耗尽（3/3）。请勿重新尝试读取。可选路径：
> 1. 利用已获取的信息进行逻辑推导
> 2. 使用 `project_info` 按需查询符号位置
> 3. 申请配额扩展（将扣除 30 衰减分）——如确需，系统会识别你的扩展请求
> 4. 直接进行实质性代码修改（edit）

**占位符加噪（防格式自噬）：** 某些模型有"强吞噬性"——读到报错里的示例格式后，会机械式地原封不动复述一遍（包括占位符 `N` 都不改），触发反滥用扣分。**拦截器返回的报错指引中严禁出现模型可以直接提取并复制的格式文本。** Quota Request 的触发不靠文本匹配 `[Quota Request: ...]`，而是由独立的系统层 token（如 `QUOTA_EXTENSION_REQUEST`）映射——模型输出的自然语言"我需要申请额外配额"被解析器识别为扩展请求，而非匹配正则。模型无法通过复制粘贴触发机制。

### 2.6 注入与拦截位置

- **注入**：每轮 LLM 调用前，根据 `token.EstimateTokens(messages) / tokenBudget` 计算水位，格式化配额消息注入上下文
- **拦截**：工具执行前检查配额计数器，超额则返回错误消息（指明可用 Quota Request 扩展）
- **检测**：LLM 返回后解析文本，发现 `[Quota Request: ...]` 则重置计数器

### 2.7 事务机制：用户中断时的状态回滚

分析大项目时 Agent 可能正在执行复杂的查询或触发强制沉淀，此时用户突然输入新指令或点击 Stop。如果配额计数器、衰减分已经在本轮扣除了但 LLM 响应被中断——模型没来得及输出总结或执行编辑——分数就白白蒸发了。

**所有扣分和配额更新必须随 LLM 成功响应而提交。** 捕获到 `User_Interrupt` 或 `Context_Cancel` 时，执行 Rollback——将配额（UsedReads/UsedGreps）和 Exploration Score 恢复到本轮调用前的快照值。独立 Merge Session 也一并取消。

**回滚边界——拦截器审计计数器绝不回滚：** 配额和分数属于"LLM 虚拟世界"的状态，可以回滚。但拦截器的内部审计计数器——文件连续读取次数、Dedup 映射——属于"物理世界"的客观事实。如果模型连续 2 轮尝试读取 `auth.go` 触发了 4.3 分页死循环检测，用户在第 3 轮打断，回滚后如果连"该文件已被读取 N 次"的记录也清空了，模型下一轮会毫无代价地继续死磕同一个文件，利用用户的打断卡死在幻觉循环里。

**回滚分两层：**
- **回滚**：`UsedReads`、`UsedGreps`、`Exploration Score`（本轮未提交的扣减撤销）
- **不回滚**：文件读取次数计数器、Dedup 映射、分页循环检测计数器（物理事实不可撤销）

**Merge 事务锁——防止回滚与异步合并的数据不一致：** 存在一个时序剪刀差：强制沉淀产出了 Summary，独立 Merge Session 正在异步执行 RPC（耗时 ~1s）。此时用户打断触发回滚——如果回滚将上下文复位，而 1s 后 Merge 成功写入了 Archive，就会导致主 Session 记忆（已回滚到旧状态）与 Archive（已被异步更新）发生数据断裂。应对：

- 启动独立 Merge Session 时，主上下文状态进入 `PendingCommit`。
- 用户中断时回滚逻辑检测该锁：**若 Merge Session 已发出，不回滚，而是以当前已产出的 Summary 强行提交。** 宁可让上下文以本轮结果为准，也不能让物理存储与上下文状态不同步。

## 三、Exploration Score — 衰减分探索槽

### 3.1 核心概念

不用死板的"连续 N 轮无行动就截断"，而是给模型一个 100 分的探索槽。每调用一次探索工具就扣分，一旦执行编辑就回满。这样高效探索能多看几轮，原地打转会快速耗尽。

### 3.2 扣分规则

| 工具 | 条件 | 扣分 |
|------|------|------|
| glob | 每次 | -5 |
| grep | 每次 | -10 |
| web_search | 每次 | -10 |
| read | 首次读此文件 | -20 |
| read | 第2次读同一文件 | -40 |
| read | 第3+次读同一文件 | 递增惩罚（-60、-80...） |
| edit / write | 修改了实际代码逻辑 | **回满 100 分** |
| edit / write | 仅修改注释/空行/格式化 | **不回分，扣 10 分** |
| Quota Request | 每次成功申请 | **扣 30 分** |

设计意图：
- glob 几乎零成本，扣很少
- read 新文件带来信息增量，扣中等
- 重复读同一文件说明在原地打转，重罚
- edit/write 意味着在干活了——但必须是真的干活

### 3.3 防伪编辑检测（Fake Edit Prevention）

模型可能采取"刷分"策略：快没分时在一个无关文件里加个空行或改个注释，触发 edit 白嫖 100 分。应对方案：

- 对比编辑前后的文件内容，判断修改是否仅涉及注释行或纯空白变更
- 如果编辑未改变任何 AST 有效节点（非注释、非空行），视为"伪编辑"——不仅不回满，额外扣 10 分
- 写入新文件不在此限制内（新建文件本身就是实质性行动）

**文件类型白名单（防误判）：** AST 差异对比只对 `.go` 文件生效。对于非 AST 文件采用不同判定规则。

**Go AST 判定必须包容类型声明变更：** 大项目重构的高频操作是改 interface 签名或 struct 字段名（如 `Host string` → `HostName string`）。这种修改常常分两步——第一轮改定义，下一轮才改实现。如果只改了 `GenDecl`（type/struct/interface 声明）而未改函数体，AST 检测可能误判为"伪编辑"并扣分，导致模型被扣死在强制沉淀里，失去继续修正实现的机会。**`GenDecl` 节点的变更必须认定为实质性编辑，立即回满分。**

**GenDecl 刷分防护——编辑去重与冷却计数器：** 但模型可能利用这个豁免规则作弊——快没分时反复在一个结构体里增删无用字段，或给 interface 添加空方法再删掉，白嫖 100 分。引入去重冷却：

- 同一文件同一 `GenDecl` 节点发生"往复修改"（A→B→A），第二次触发时不回满分，罚扣 30 分。
- 只有当 `GenDecl` 变更伴随其他文件的引用变化（如 `grep` 到该符号被引用，或其他文件触发了 `FuncDecl` 变更）时，才认定为有效重构并回满。
- 计数器随有效编辑（含其他文件的关联变更）重置。

| 文件类型 | 判定方式 | 有效编辑条件 |
|----------|----------|-------------|
| `.go` | AST 差异对比 | 非注释、非空行的 AST 节点变更，**含类型声明变更** |
| `.mod`, `.sum` | 行级 Diff | 删减或新增行数 > 0 |
| `.yaml`, `.json`, `.toml` | 词法符号变动 | 有效键值对的新增、修改或删除 |
| `.md` | 行级 Diff | 删减或新增行数 > 2（防改一个词刷分） |
| 其他未知类型 | 行级 Diff | 变更行数 > 2 |

没有 AST 的文件（配置、文档、依赖声明）在大型重构中间同样至关重要——改 `go.mod` 升级依赖、改 `.yaml` 调配置，这些必须是合法行动。

### 3.4 止损方式：强制沉淀而非截断

分数扣完时不粗暴中断，而是注入：

> 探索槽已耗尽（上下文碎片化）。请立即停止读取类工具调用，用结构化 summary 总结当前所有技术线索。系统收到总结后自动 GC 无用历史 + 重置探索预算。

代码侧动作：
1. 注入上述提示，等待模型输出文本
2. 模型输出总结 → 触发 MicroCompact + ForceCompact 做局部 GC → 重置探索槽 → 注入绿区配额
3. 模型无视提示继续调工具 → 返回错误，强制要求先总结

止损的本质不是惩罚，而是强制帮它做一次局部垃圾回收，清空前几轮的冗余 read 后重新给满血配额。

### 3.5 结构化总结模版

触发止损时，不只要求模型"写个总结"——而是要求按固定的 key-value 模版输出，方便后续 Memory 系统收割，也迫使模型把"读了一堆代码"收敛到"能说清模块职责和依赖"：

```
<module-summary>
  Module_Path: [相对路径]
  Main_Responsibility: [一句话职责描述]
  Key_Dependencies: [逗号分隔的依赖路径或接口]
  State: [Explored / Needs_Refactor / Stable]
</module-summary>
```

每个被分析过的子模块都产出这样一条记录。这些记录被客户端收集后，写入 Memory 或拼回 Layer 0.5 Archive。

### 3.6 总结去重与合并

如果每次强制沉淀都产出一批 `<module-summary>`，多次沉淀后 Layer 0.5 会被重复的总结文本撑爆。应对：

- 客户端收集到 `<module-summary>` 后，**按 Module_Path 去重**
- 同一路径的模块只保留最新一条——但**不是简单覆盖，而是增量合并（Incremental Merge）**
- 合并时将旧总结作为上下文传入，模型需要判断：哪些旧信息仍然有效、哪些已被新探索推翻、哪些需要保留为历史注记
- 合并后的精简集才写入 Memory/Archive

**合并的执行主体——不能用当前 LLM：** 触发强制沉淀时 Exploration Score 已归零，说明当前上下文碎片化严重。让此时已经"脑满肠肥"的主 LLM 来做增量合并，它大概率无法高质量完成，甚至可能直接无视 Merge 格式导致止损失败。

正确做法：在代码侧，将"旧总结"和"本轮新产出的总结"单独打包，**开辟一个干净的、无历史包袱的独立 LLM Session**（不带任何历史消息，只发一条 Merge 提示词）专门执行合并。参数精简（thinking=disabled, max_tokens=2000），耗时 < 2s。

**合并失败兜底（防主线程阻塞）：** 独立 Merge Session 必须配置独立的指数退避重试 Client。高并发或服务端 Rate Limit 可能导致合并请求失败——此时**绝不能挂起用户主会话**。兜底策略：如果重试后仍失败，直接将新旧总结进行物理字符串拼接，打上 `[Merge Failed — raw append]` 标签写入 Archive。先保证系统继续运行，一致性下次沉淀时修复。

**异步自愈工作流（防连续熔断致 Archive 降级断层）：** 如果线上服务持续抖动，连续 3~4 次强制沉淀都遇到独立 Merge 失败，Archive 内部会堆积多个未被收敛的、冗长且包含矛盾逻辑的 RAW 拼接文本。当下一次 Merge Session 终于成功拉起时，它面对的不是精简上下文，而是被物理拼接撑爆的碎片化 Archive——轻量模型（max_tokens=2000）直接上下文爆仓，导致合并永久锁死在失败状态。

**终期防线：**
- 当检测到 Archive 中存在 `[Merge Failed — raw append]` 标签时，不仅要完成当前写入，还要异步向后台任务队列投递一个**低优先级自愈任务**。
- 该任务完全独立于用户交互线程运行，使用更大窗口的模型 + 更长超时（如 10s），专门对带 `[Merge Failed]` 的 Archive 做强力清洗与收敛。
- 清洗完成后平滑替换物理 Archive 文件。确保下一次主线程触发沉淀时，面对的永远是干净健康的知识集。

**为什么不能简单覆盖：** 模型在第 5 轮探索了 `auth/service.go` 并生成了总结。第 20 轮代码已被重构，内部逻辑发生变化。如果直接 Overwrite 旧总结，会丢失前 20 轮沉淀下来的、当前轮次没有重新读到的关键历史细节。增量合并确保历史知识不丢失，只更新变化的部分。

## 四、Tool-Result Interceptor — 工具结果拦截器

### 4.1 修复 Budget 截断（已有代码但未生效）

`compact/budget.go` 已经实现了按工具类型的字符限制（read/bash 6000字符、grep/glob 2000字符），但 `storage.go` 调用时 toolName 传的是空字符串，导致 budgetLimit("") 返回 0（无限制），所有结果都绕过截断。

修复：传递真实 toolName，让已有的预算截断逻辑生效。

### 4.2 Read 工具强制分页

Read 工具已经有 `startLine`/`endLine` 参数，但当模型不指定行号直接读大文件时，不会主动拦截。需要加入自动截断：

- 文件超 500 行且没指定 startLine/endLine → 只返回前 200 行
- 同时附带警告元数据：`[文件共 N 行，未指定行号时自动截断前 200 行。如需后续内容请指定 startLine/endLine 精确读取]`

### 4.3 分页死循环检测

强制分页可能引发另一种灾难：模型陷入"接力棒式"循环——读 1-200 → 被截断 → 读 201-400 → 被截断 → 读 401-600... 反而加速 token 消耗。

应对：当同一文件被连续分页读取 **2 次以上** 时，Interceptor 主动介入，不再返回原始内容，而是用本地工具（grep -n 提取函数/类型签名）生成该文件的"结构大纲"，直接喂给模型：

```
[文件 {path} 共 N 行。已连续分页读取 2 次，触发防循环保护。]
文件结构大纲:
  L12:  import (...)
  L25:  type Config struct { ... }
  L48:  func New() *Service { ... }
  L92:  func (s *Service) Process(ctx Context) error { ... }
  ...

如需查看具体实现，请指定函数名而非行号范围。
```

这样模型看到的不是碎片化的原始代码，而是文件骨架——帮助它理解结构而无需逐行吞入。

### 4.4 Dedup 去重

维护一个本轮次 read 记录映射。当同一文件被读取多次时：
- 范围完全匹配（同一路径+同一行号区间）→ 返回指针 "内容同第 N 次 read 结果"，不重复注入
- 同一文件不同位置被读超过 3 次 → 缩短返回并提示"该文件已被读取多次，如需特定段落请指定精确行号"

注意与旧 UNCHANGED stub 的区别：旧方法是隐藏内容只给 800 字符预览，导致模型失去上下文。新方法保留首次完整结果在上下文中，仅在重复时给指针——模型始终能找到完整内容。

## 五、ProjectIndex — 静态分析预计算

### 5.1 问题

每次对话启动，模型对项目结构一无所知，浪费 3-5 轮做最基本的探索——有哪些包、哪些文件、依赖关系如何。这部分成本是纯浪费，因为项目结构在会话期间不会变。

### 5.2 分层注入策略（防膨胀）

一个 500 文件、3000 符号的中型项目，全量 `<project-index>` 可能吃掉 20k-50k token。如果每轮都塞在 Layer 0 中，启动成本虽然省了，但日常轮询成本暴增。

**分级注入：**

- **Layer 0（每轮携带）**：只注入第一层目录骨架 + 核心依赖包列表（约 500-1000 token）。比如"项目有 12 个包，bot/agent → bot/ctxmgr, llm..."
- **Layer 0.5 / 查询工具（按需）**：详细符号索引（3000 个 func/type）不注入上下文，而是做成 `project_info` 本地只读工具

### 5.3 project_info 工具（从可选提升为必需）

`project_info` 不是"后续可选增强"，而是 **Phase 1 必须实现**——它替代了"把整张地图贴在模型脸上"的粗暴方案：

- `project_info(query="symbol:Agent")` → 返回 `type Agent struct` 在 `bot/agent/agent.go:68`
- `project_info(query="deps:bot/agent")` → 返回依赖和被依赖列表
- `project_info(query="file:manager.go")` → 返回文件所在包、行数、导出符号

模型需要哪个模块的信息就查哪个，按需索取，而非一次性吞下整个索引。
- 在全量索引之上分层：Layer 0 只放骨架，详细符号做成 `project_info` 工具按需查询
- 构建时用 `go list -json ./...` 获取包列表和导入关系，但**必须过滤到仅内部包**

**go list 的依赖爆炸防护：** `go list -json ./...` 会拉出包括 `net/http`、`runtime`、数万行第三方依赖的全量依赖树。不加以过滤，索引构建会吃掉数百兆内存，击穿 < 2s 阈值。必须强制加入内部包过滤器：

- **内部包**：路径前缀匹配当前 `go.mod` 的 Module Name——只追踪项目自身包之间的调用拓扑
- **外部依赖**：仅记录到第一层（如 `bot/agent → github.com/gin-gonic/gin`），绝不向下递归扫描第三方库内部的依赖图
- **标准库**：完全排除（`fmt`、`net/http`、`context` 等），模型不需要知道这些依赖关系
- 缓存到 `~/.nekocode/index/{hash}.json`

**缓存键必须包含 Git 状态（防分支切换失效）：** 程序员经常在不同 Git 分支之间切换。两个分支的 `go.mod` + `go.sum` 可能完全一样，但业务源码已天翻地覆。仅按 mod 哈希索引，切分支后 Agent 会读到旧分支的过期缓存，导致符号定位全面错误。缓存键公式：

```
CacheKey = Hash(go.mod + go.sum + GitHEAD_CommitID + UncommittedFilesStatus)
```

其中 `UncommittedFilesStatus` 是 `git status --porcelain` 的输出哈希——确保即使在同一 commit 上，本地未提交的修改也会触发索引更新。缓存键变化时不是全量重建，而是比对文件 ModTime，只重新扫描变更文件。

**正则扫描的 Monorepo 防护：**

- **Go 生成代码排除**：默认跳过 `*.pb.go`（Protobuf）、`*_mock.go`（mock）、`*_gen.go`（通用生成）、`testdata/` 目录
- **全局目录黑名单（防异构项目 I/O 阻塞）**：大型项目往往是混合语言的——Go 后端里可能包含前端 `node_modules/`、Python 脚本、Rust 的 `target/` 等。无论什么语言，只要路径命中以下模式，一律从文件流层面过滤，不参与哈希计算和磁盘轮询：

```
node_modules/  vendor/  target/  .git/  dist/  build/
__pycache__/  .cache/  .next/  .turbo/  coverage/
*.min.js  *.min.css  *.pb.go  *_mock.go  *_gen.go
```

**ModTime 增量扫描**：按单文件 `ModTime` 做增量更新。大仓库下首次扫描后，后续只重新扫描有变更的非黑名单文件。

**惊群效应防护——Single-Flight 构建队列：** 如果 1000 个文件被 `gofmt` 格式化了，所有 ModTime 刷新，增量扫描瞬间退化为全量 I/O。此时用户连续发起多个对话或多个分析任务并发启动 → 疯狂并发 `go list` + 正则扫描 → CPU 和磁盘 I/O 瞬间拉满。

- **Single-Flight**：同一项目的索引构建用 `golang.org/x/sync/singleflight` 保护。缓存失效且正在重构时，后续并发请求直接阻塞并复用同一个构建结果，绝不重复扫描。
- **并发限流**：正则扫描使用信号量限制并发数不超过 CPU 核心数的 50%，防止 I/O 饥饿。

## 六、可观测性 — Agent Context Trace

### 6.1 问题

方案包含配额、衰减分、拦截器、强制分页、大纲保护、事务回滚、独立 Merge 等多个精密咬合的机制。一旦线上模型突然不动或分数莫名归零，开发人员根本无法分清是哪个机制触发了干预——排查变成猜谜。

### 6.2 标准化 Trace 日志

从 Phase 1 落地起，每一次工具调用前后必须输出标准化的 Trace 日志：

```
[NekoCode-Trace] Turn:5 | Watermark:52%(Yellow) | Score:70→50(Read:NewFile) | Quota:Read(2/3) Grep(1/2)
[NekoCode-Trace] Turn:5 | Score:50→10(Read:ReRead×3) | Quota:Read(3/3)
[NekoCode-Trace] [Interceptor] Loop detected: file=auth.go, 2 sequential reads → downgrading to outline
[NekoCode-Trace] Turn:6 | Score:10→100(Edit:GenDecl) | Quota reset
[NekoCode-Trace] Turn:8 | [QuotaExtend] Granted +1 Read | Score:80→50(Penalty:-30)
[NekoCode-Trace] Turn:10 | Score:0→ForcedPrecipitate triggered
[NekoCode-Trace] [MergeSession] Success (1.2s, 342 tokens) → Archive merged
[NekoCode-Trace] Turn:5 | [Rollback] UserInterrupt detected → Score restored to 70, Quota restored
```

每行 Trace 必须包含：轮次、水位、分数变化（+原因）、配额状态、触发的机制名称。确保在复杂的线上大项目分析中，每一层干预都清晰可见。

## 实现顺序与优先级

**原则：先给工具，再加限制。** 配额和扣分上线前，模型必须先有 `project_info` 作为不扣分的导航替代手段。否则限制层一上，模型没有逃生通道，直接瘫痪。

| 阶段 | 内容 | 优先级 | 说明 |
|------|------|--------|------|
| Phase 1 | 修复 budget.go 传 toolName | P0 | 已有代码，只修 bug |
| Phase 1 | ProjectIndex + project_info 工具 | P0 | 必须先于配额上线，提供不扣分的导航替代 |
| Phase 2 | Dedup 拦截器 + Read 强制分页 | P1 | 上下文空间治理 |
| Phase 3 | 配额系统（quota.go + 事务回滚） | P1 | 按水位动态限制，含中断回滚 |
| Phase 4 | 衰减分（exploration.go + 伪编辑检测） | P1 | 含文件类型白名单 |
| Phase 5 | 系统提示词更新 | P1 | 编排决策树 + Manifest-First + 配额扩展说明 |
| Phase 6 | 总结合并增量 + 独立 Merge Session | P2 | 依赖 Phase 4 的强制沉淀触发 |
